# Obsidian project context

Use the project’s Obsidian documentation folder as the first source for architecture, API, features, and conventions. Reduces repeated codebase scans and keeps context in one place.

## When to use

- You need: stack, entry points, routes, API endpoints, critical files, naming/conventions.
- The project has (or may have) an Obsidian docs folder (e.g. generated by **project-docs-db**).
- You want to avoid re-scanning the whole repo for “how this project is structured”.

**When not to use:** No Obsidian docs for this project; or you need line-level details in a specific file (read the file instead).

## Finding the Obsidian docs path

1. **Config in repo** — Check for a path stored in the project:
   - `.cursor/obsidian-docs-path` (one line: full path to the project’s docs folder in the vault)
   - Or a path mentioned in `.cursor/rules`, README, or SETUP_GUIDE.
2. **Convention** — If the project was documented with project-docs-db, the folder is often:
   - `{vault}/work/projects/{project-name}/`
   - Example: `C:\Users\...\Obsidian Vault\work\projects\password-manager`
3. **Ask** — If unclear, ask the user: “Where is this project’s Obsidian docs folder (path inside your vault)?”

## Core pattern

1. **Before** diving into code for “how is this built?” or “what endpoints exist?”:
   - Resolve the Obsidian docs path (config / convention / user).
   - Read the relevant note(s) with the Read tool (use the full path to the `.md` file).
2. **Use docs for:** stack, project map, feature list, routes, API surface, critical files, commands, gotchas.
3. **Use code for:** exact implementation, specific components, debugging, or when docs are missing/outdated.

## Quick reference

- **One-file summary:** `agent-context.md` (best first read).
- **Path in this repo:** If present, see `.cursor/obsidian-docs-path`; otherwise use vault path + `work/projects/<project-name>/`.
- **Files are Markdown:** Read with the normal file Read tool; no special MCP required.

## Common mistakes

- **Re-scanning the codebase** when architecture/API are already described in Obsidian — check for an Obsidian docs path first.
- **Assuming the path** — If there’s no `.cursor/obsidian-docs-path` or obvious convention, ask for the path once and optionally suggest adding it to `.cursor/obsidian-docs-path`.
- **Treating docs as always up to date** — For exact types or new code, prefer the codebase; use docs for orientation and then open the referenced files.

## Optional: persist the path

To avoid asking every time, the project can store the Obsidian docs path:

- Create `.cursor/obsidian-docs-path` (or add to a rule) with one line: the full path to the project’s docs folder inside the vault.
- Example content: `C:\Users\USER\OneDrive\Desktop\Documents\Obsidian Vault\work\projects\password-manager`

Then the agent can read that file first to resolve the path.
